# PRD: Веб-приложение для оценки бизнес-идей

## 1. Контекст и цель
Внутреннее веб-приложение для сбора, фильтрации и проработки бизнес-идей, получаемых из внешних источников (в первую очередь из фриланс-бирж).  
Система должна поддерживать поток:
- автоматическая загрузка идей внешними скриптами,
- быстрый первичный отбор (да/нет),
- детальная проработка отобранных идей,
- обзор воронки и текущего прогресса на главной странице.

## 2. Область первой версии (MVP)
### Включено
- Backend: Django + DRF.
- Frontend: React.
- БД: PostgreSQL.
- Инфраструктура запуска: Docker Compose.
- Ingestion API для приема идей (по одной или пачкой).
- Дедупликация по паре (`source_system_id`, `source_id`) с игнорированием дублей.
- Экран быстрой фильтрации (красная/зеленая кнопка, в стиле swipe-механики).
- Экран(ы) проработки карточек идей с редактированием полей.
- Главная страница с мини-дашбордом (воронка по статусам).
- Django Admin для служебного управления и физического удаления записей.

### Не включено
- Любая пользовательская сущность и аутентификация.
- Soft delete / архивная модель удаления.
- Встроенные парсеры как часть этого репозитория (скрипты внешние).

## 5. Жизненный цикл идеи (статусы)
Единая статусная модель:
- `new`
- `accepted`
- `rejected`
- `in_progress`
- `completed`

Правила переходов:
1. Новые идеи создаются со статусом `new`.
2. На экране фильтрации:
   - красная кнопка: `new -> rejected`;
   - зеленая кнопка: `new -> accepted`.
3. На этапе проработки, если `is_filled = false` и заполнено хотя бы одно поле из списка проработки, статус автоматически становится `in_progress`.
4. Статус `completed` устанавливается только вручную через `is_filled = true`.
5. При `is_filled = true` фиксируется `filled_at` (если не задана ранее) датой по умолчанию = текущая дата.

## 6. Модель данных
## 6.1 Таблица `source_system`
Справочник источников.
Поля:
- `id` (int, PK)
- `name` (string)
- `base_url` (string, optional)
- `is_active` (bool)
- `created_at`, `updated_at`

## 6.2 Таблица `stage`
Основная таблица идей.
Поля:
- `id` (bigint, PK)
- `source_system_id` (int, FK -> `source_system.id`) — обязательное
- `source_id` (varchar(512)) — обязательное
- `source_url` (text) — обязательное
- `title` (text) — обязательное
- `description` (text, nullable)
- `category` (text, nullable)

Поля проработки:
- `custom_title` (text, nullable)
- `custom_description` (text, nullable)
- `existing_solution` (text, nullable)
- `original_revenue_estimate` (text, nullable)
- `seo_query` (text, nullable)
- `seo_kd_percent` (int, nullable, 0..100)
- `seo_popularity_vs_adblocker` (text, nullable)
- `planned_feature` (text, nullable)
- `implementation_ease_percent` (int, nullable, 0..100)
- `risks` (text, nullable)

Служебные поля:
- `status` (enum: `new|accepted|rejected|in_progress|completed`)
- `is_filled` (bool, default `false`)
- `filled_at` (date, default current date при установке `is_filled=true`)
- `created_at`, `updated_at`

Ограничения:
- Уникальный индекс: (`source_system_id`, `source_id`).
- CHECK для процентов: `seo_kd_percent` и `implementation_ease_percent` в диапазоне 0..100.
- Индексы для производительности (десятки тысяч записей):
  - `status`
  - `created_at`
  - `is_filled`
  - составной индекс (`status`, `created_at`)

Требование расширяемости:
- Структура и формы должны быть организованы так, чтобы можно было относительно просто добавлять новые поля карточки (и в API, и в UI).

## 7. API требования (DRF)
## 7.1 Ingestion endpoint
Назначение: прием данных от внешних парсеров.

Формат:
- Прием одного объекта и массива объектов.
- Минимально обязательные поля для записи:
  - `source_system_id`
  - `source_id`
  - `title`
  - `source_url`
- Опциональные: `description`, `category`.

Поведение дедупликации:
- Если запись с парой (`source_system_id`, `source_id`) уже существует, входящий объект игнорируется.
- Обновление существующей записи при дубле не выполняется.

Ответ:
- Возвращать количество созданных и проигнорированных записей.

## 7.2 API для UI
Нужны endpoint’ы для:
- списка идей с фильтрацией/сортировкой/пагинацией;
- получения одной идеи;
- смены статуса на этапе фильтрации (`accepted`/`rejected`);
- обновления карточки проработки;
- получения агрегатов для дашборда (воронка по статусам).

Требования к спискам:
- Пагинация обязательна (целевой объем: десятки тысяч).
- Дефолтная сортировка: `created_at` по возрастанию.

## 8. UI/UX требования
Общее требование к локализации:
- Пользовательский интерфейс MVP должен быть выполнен на русском языке.
- Навигация, кнопки, заголовки, подписи полей, статусы и пользовательские сообщения об ошибках/загрузке должны отображаться на русском языке.
- Допускается использование английского только для технических идентификаторов, API-контрактов, названий технологий и служебных URL.

## 8.1 Главная страница (Dashboard)
Содержимое:
- краткая воронка по статусам (`new`, `accepted`, `rejected`, `in_progress`, `completed`),
- базовые счетчики и простая визуализация прогресса,
- навигация к двум рабочим этапам.

## 8.2 Страница 1: Быстрая фильтрация (Да/Нет)
Сценарий:
1. Показывается одна идея: `title`, `description`, `category`, `source_url`.
2. Пользователь нажимает:
   - красную кнопку -> идея получает `rejected`;
   - зеленую кнопку -> идея получает `accepted`.
3. Следующая идея подгружается динамически до конца доступного списка.

Правила отображения:
- По умолчанию показываются идеи, требующие первичной оценки (прежде всего `new`).
- `rejected` по умолчанию скрыты, но в интерфейсе должна быть возможность включить их просмотр отдельным фильтром.

## 8.3 Страница 2: Этап проработки (несколько экранов/секций)
Сценарий:
1. Пользователь выбирает идею из списка (обычно из `accepted`).
2. Заполняет/редактирует карточку проработки.
3. Сохраняет промежуточный результат.

Логика статуса:
- Если заполнено хотя бы одно поле проработки и `is_filled = false`, автоматически выставляется `in_progress`.
- Когда пользователь вручную ставит `is_filled = true`, статус становится `completed`.

Требования к списку/панели:
- Мини-панель фильтров и сортировки.
- Фильтры минимум:
  - по статусу,
  - по источнику,
  - по категории,
  - по заполненности (`is_filled`),
  - возможность видеть `rejected`.
- Сортировка по умолчанию: дата создания по возрастанию.
- Заполненные карточки должны визуально выделяться цветом.

## 9. Нефункциональные требования
- Целевой объем данных: десятки тысяч записей.
- Серверная пагинация для всех больших списков.
- Индексация и оптимизация запросов по статусам и времени создания.
- Базовая валидация входных данных API.
- Логирование ingest-операций (создано/игнорировано/ошибки).

## 10. Технологические требования
Backend:
- Django
- Django REST Framework
- PostgreSQL

Frontend:
- React
- UI приложения для конечного пользователя должен отображаться на русском языке.

Инфраструктура:
- `docker-compose.yml` минимум для сервисов:
  - `web` (Django API)
  - `db` (PostgreSQL)
  - `frontend` (React)
- Приложение разворачивается и используется локально на компьютере разработчика (localhost).
- Для локального запуска используются адреса по умолчанию:
  - backend API: `http://localhost:<backend_port>`
  - frontend: `http://localhost:<frontend_port>`

Важно:
- Парсеры запускаются отдельно от этого docker-compose и взаимодействуют с приложением только через ingestion API.

## 11. Администрирование
- Django Admin включен для внутреннего обслуживания данных.
- Разрешено физическое удаление объектов через админку.
- Soft delete не требуется.

## 12. Критерии приемки MVP
1. Можно загрузить идеи через API по одной и пачкой.
2. Дубликаты по (`source_system_id`, `source_id`) не создаются и игнорируются.
3. Экран фильтрации позволяет последовательно проставлять `accepted/rejected` и подгружает следующую идею без перезагрузки всей страницы.
4. `rejected` хранятся в БД, скрыты по умолчанию, доступны через фильтр.
5. Экран проработки позволяет редактировать все заявленные поля.
6. Автоматически выставляется `in_progress`, если начато заполнение карточки при `is_filled=false`.
7. `completed` выставляется только при ручной установке `is_filled=true`.
8. Заполненные карточки визуально выделяются и фильтруются.
9. Главная страница показывает мини-воронку по статусам.
10. Проект запускается через Docker Compose (backend + frontend + postgres).

## 13. Требования к тестированию
### 13.1 Цель
В проекте обязательно должна быть автоматизированная проверка ключевой бизнес-логики и пользовательских сценариев на трех уровнях:
- backend-тесты (Django/DRF),
- frontend-тесты (React),
- end-to-end тесты (сквозные сценарии в браузере).

### 13.2 Стратегия покрытия (тестовая пирамида)
- Основной акцент: backend-тесты (бизнес-правила, API, валидации, фильтры, пагинация).
- Второй слой: frontend-тесты (поведение экранов и компонентов с моками API).
- Верхний слой: e2e-тесты (ограниченный набор критичных пользовательских потоков).

Рекомендованное распределение:
- ~60% backend,
- ~30% frontend,
- ~10% e2e.

### 13.3 Рекомендуемый стек
Backend:
- `pytest`
- `pytest-django`
- DRF test client (`APIClient`)
- `factory_boy` (или эквивалент) для подготовки тестовых данных

Frontend:
- `Vitest` (или `Jest`)
- `React Testing Library`
- `MSW` для мокирования API

E2E:
- `Playwright`

### 13.4 Обязательные backend-сценарии
1. Ingestion endpoint:
   - прием одного объекта и массива,
   - корректный подсчет `created/ignored`,
   - дедупликация по (`source_system_id`, `source_id`) без обновления дублей.
2. Валидации:
   - обязательные поля ingestion,
   - диапазон `0..100` для `seo_kd_percent` и `implementation_ease_percent`.
3. Логика статусов:
   - `new -> accepted/rejected` на этапе фильтрации,
   - автоматический `in_progress` при заполнении хотя бы одного поля проработки при `is_filled=false`,
   - `completed` только при ручной установке `is_filled=true`,
   - фиксация `filled_at` при первом переводе `is_filled=true`.
4. Списки и агрегации:
   - серверная пагинация,
   - сортировка по умолчанию `created_at ASC`,
   - фильтры по статусу/источнику/категории/`is_filled`,
   - агрегаты для дашборда (воронка по статусам).

### 13.5 Обязательные frontend-сценарии
1. Экран быстрой фильтрации:
   - нажатие красной/зеленой кнопки вызывает корректный API-запрос,
   - после ответа отображается следующая идея без полной перезагрузки.
2. Отображение `rejected`:
   - `rejected` скрыты по умолчанию,
   - включаются отдельным фильтром.
3. Экран проработки:
   - сохранение изменений в полях карточки,
   - корректное отображение статуса `in_progress/completed`,
   - визуальное выделение заполненных карточек.
4. Дашборд:
   - корректное отображение счетчиков и воронки из API.

### 13.6 Обязательные e2e-сценарии (минимум)
1. Сквозной ingestion:
   - загрузка пачки идей,
   - дубликаты не создаются.
2. Быстрая фильтрация:
   - последовательное проставление `accepted/rejected` для нескольких карточек,
   - динамическая подгрузка следующей карточки.
3. Проработка:
   - заполнение полей -> переход в `in_progress`,
   - установка `is_filled=true` -> переход в `completed` и фиксация `filled_at`.
4. Фильтры и дашборд:
   - проверка скрытия/показа `rejected`,
   - согласованность агрегатов дашборда с фактическими статусами.

### 13.7 Запуск тестов в локальной среде
- CI не является обязательным требованием MVP.
- Все тесты должны запускаться локально на компьютере разработчика в окружении `docker-compose`/`localhost`.
- Рекомендуемый порядок локального прогона:
  1) backend tests,
  2) frontend tests,
  3) e2e tests.

### 13.8 Критерии приемки по тестированию
MVP считается принятым только при выполнении одновременно всех условий:
1. Есть автоматизированные backend, frontend и e2e тесты.
2. Покрыты обязательные сценарии из разделов 13.4-13.6.
3. Все тесты проходят локально в целевой конфигурации проекта (`docker-compose`/`localhost`).
