# Декомпозиция проекта на задачи

## Принципы
- Одна задача должна укладываться в один рабочий чат.
- После каждой задачи проект остаётся запускаемым через `docker compose up`.
- У каждой задачи в конце есть проверка: тест, smoke-check или ручной сценарий.
- Задачи идут в порядке, который минимизирует блокировки между backend, frontend и инфраструктурой.

## Как читать этот файл
- `ID` — короткий идентификатор задачи.
- `Зависит от` — что должно быть сделано раньше.
- `Проверка` — чем закрывается задача.
- `Runnable после задачи` — почему после неё проект всё ещё можно поднять и использовать.

## Правило отчётности по задачам
- После завершения каждой задачи бот обязан дописать отчёт в файл `task-reports.md`.
- Один завершённый таск = один новый блок отчёта.
- Старые записи не перезаписываются; история выполнения должна сохраняться.
- Отчёт пишется только после того, как код внесён и проверка задачи выполнена.

### Шаблон отчёта
```md
## T17 — Реализовать ingestion endpoint для одного объекта
- Дата: 2026-03-08 14:20 +03:00
- Статус: done
- Коммит: `abc1234`

Что сделано:
- добавлен POST endpoint ingestion;
- добавлено создание `Stage` со статусом `new`;
- добавлен backend-тест на успешное создание.

Проверка:
- `pytest backend/app/tests/test_ingest.py` — passed
- smoke: `POST /api/ingest/` — passed

Состояние запуска:
- `docker compose up` проходит
```

### Обязательные поля отчёта
- идентификатор и название задачи;
- дата завершения;
- статус;
- коммит;
- краткое описание выполненной работы;
- список проверок и их результат;
- подтверждение, что проект остаётся запускаемым.

## Definition of Done для одной задачи
- Объём задачи помещается в один рабочий чат без разрастания в мини-эпик.
- После выполнения проект всё ещё поднимается через `docker compose up`.
- Есть тест, smoke-check или формализованная ручная проверка.
- После выполнения дописан отчёт в `reports/task-reports.md` по шаблону из этого файла.
- Если менялась схема, добавлены миграции.
- Если менялись команды запуска или тестирования, обновлён `README.md`.

## Что стоит ещё сильнее дробить при необходимости
- T40 можно разделить на 2-3 задачи по секциям формы, если полей окажется слишком много для одного чата.
- T39 можно разделить на отдельные задачи по фильтрам, если UI-архитектура будет сложной.
- T20 можно объединить с T19, если логирование окажется тривиальным.
- T52 можно отложить до самого конца, если сначала важнее закрыть весь обязательный функционал.
